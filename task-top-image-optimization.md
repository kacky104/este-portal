# タスク：トップページの画像を next/image 化（初回訪問者の転送量削減）

## 目的
トップページに表示される画像が素の `<img>` タグで Supabase Storage の原寸URLを直接配信しており、初回訪問者（キャッシュ空）への総転送量が約34MBと過大。これを next/image に置き換え、自動リサイズ・WebP変換・遅延読み込みを効かせて、転送量と表示速度を大幅に改善する。

**今回のスコープはトップページに出る画像のみ。** マイページ（/mypage）の編集画面や日記詳細ページなど、トップに出ない画像は今回は対象外（次回に回す）。

## 前提（確認済み）
- `next.config.ts` の images.remotePatterns に Supabase ドメイン（efjrpanojfahqjwqpagg.supabase.co の /storage/v1/object/public/**）が登録済み。**設定変更は不要。**
- `HeaderImageSlider.tsx` で既に next/image を使用しておりビルドも通っている。**この既存実装を参考（お手本）にすること。**

## ルール
- 見た目（レイアウト・サイズ・object-fit の挙動）は変えない。置き換え前と後でピクセル単位で同じ見た目を保つ。
- DB スキーマ変更なし。環境変数の変更なし。
- 完了後に型チェック・lint・build クリーンを確認し、feedback_auto_deploy.md の手順で自動コミット→プッシュ→デプロイまで進める。

## 実装方針

### 共通：2種類の置き換えパターンを使い分ける

**パターンA：親要素いっぱいに広げる画像（背景・object-cover でカードを埋める系）**
- 例：FeaturedSalonSlider の背景画像、各カードを埋める写真。
- next/image の `fill` を使う。
  - 画像の**親要素に `position: relative`（Tailwind なら `relative`）と明示的なサイズ**が付いていることを確認する（無ければ付ける）。
  - `<Image src={...} alt={...} fill className="object-cover" sizes="..." />` の形にする。
  - 元の `<img>` が持っていた className のうち、object-fit やフィルタ系は Image 側に引き継ぐ。w/h 指定系は親要素へ移す。

**パターンB：固定サイズで表示する画像（幅・高さが決まっている小サムネイル）**
- 例：セラピストの丸い顔写真など、明確な寸法がある場合。
- `width` と `height` を明示した `<Image src={...} alt={...} width={N} height={N} sizes="..." />` にする。
- 元の表示サイズに合わせた数値を入れる。

### sizes の指定（今回の最重要ポイント）
sizes を省略すると next/image が画面幅基準で大きすぎる画像を配信し、逆に重くなる。各画像の**実際の表示幅**に合わせて指定する。目安：

- セラピストの小さい顔写真（1行に複数並ぶサムネイル）
  → `sizes="(max-width: 640px) 25vw, 120px"` 程度（スマホで画面の約1/4、PCで約120px）。
- ピックアップ／サロンカードの中〜大画像（1カラム〜数カラム）
  → `sizes="(max-width: 640px) 100vw, (max-width: 1024px) 50vw, 400px"` 程度。
- ヘッダー級の大きい1枚絵（横幅いっぱい）
  → `sizes="100vw"`。

※ 上記はあくまで目安。各カードの実際のレイアウト（何カラムか、最大幅いくつか）を見て調整すること。表示幅より少し大きめに見積もるのが安全。

### alt 属性
- 既存の `<img>` が持っている alt をそのまま引き継ぐ。空 alt（alt=""）のものは、意味のある内容（サロン名・セラピスト名など near にある変数）があれば設定する。無ければ空のままで可。

### priority（任意）
- ファーストビューに最初から見える大きい画像（例：ピックアップの1枚目、ヘッダー画像）には `priority` を付けて初期表示を速くしてよい。逆に下の方の画像には付けない（付けすぎると逆効果）。

---

## やってはいけないこと
- next.config を触らない（設定は既に正しい）。
- sizes を省略しない（省略は逆効果になりうる）。
- fill を使うとき、親要素の position: relative を付け忘れない（画像が消える/崩れる原因）。
- スコープ外（mypage 等）の `<img>` には手を付けない。

- `npm run lint`：エラーなし。

- ローカル or 本番で、トップページの見た目が置き換え前と**同じ**であること（画像が消えていない・潰れていない・位置がずれていない）。スマホ幅・PC幅の両方で確認。
- DevTools の Network（画像フィルタ）で、トップの画像が webp で配信され、1枚あたりのサイズが置き換え前より小さくなっていること。総リソース量（34MB）が大きく減っていること。
