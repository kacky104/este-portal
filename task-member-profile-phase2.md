# タスク：会員プロフィール（フェーズ2・ニックネーム）

## 目的
会員（一般ユーザー）が自分のニックネームを設定・編集できるようにする。会員ダッシュボード（/member）の「プロフィール編集（準備中）」枠を、実際に動く機能に置き換える。編集は専用ページ /member/profile で行う。

**フェーズ2のスコープはニックネームのみ。** アバター画像・好きなエリア等は今回やらない（将来 /member/profile に列とUIを足して拡張する）。

## 前提（重要）
- DBテーブル `public.profiles` は**作成済み**（手動SQL実行済み）。構成：
  - `id uuid`（primary key、auth.users(id) と1対1、on delete cascade）
  - `nickname text`
  - `created_at timestamptz`, `updated_at timestamptz`
  - RLS有効。本人のみ select / insert / update 可能（auth.uid() = id）。
- 会員ログインは既存の Supabase Auth。/member・/member/profile は**ログイン必須**（未ログインなら /login へリダイレクト）。
- /mypage はオーナー専用。**一切触らない。**
- ブランド色（ロゴ・見出し＝オレンジ→マゼンタ #FB923C→#DB2777）を踏襲。

## ルール
- 新規ページ追加・UI変更なので、feedback_auto_deploy.md の手順で自動コミット→プッシュ→デプロイまで進める。
- DBスキーマ変更はしない（テーブルは作成済み。コード側のみ実装）。
- 環境変数の変更なし。既存ページ（/saved・/mypage・トップ・サロン詳細等）の挙動を変えない。

## 実装内容

### 1. プロフィール編集ページ /member/profile を新規作成
- 新規ファイル `src/app/member/profile/page.tsx`。
- ログイン必須（サーバー側で Supabase Auth セッションを確認、未ログインなら /login へ redirect）。
- ログインユーザーの profiles 行を取得して、ニックネーム入力欄にプリフィルする。行が無ければ空欄。
- UI：
  - ヘッダーは既存の共通ヘッダーを流用。
  - 「プロフィール編集」の見出し。
  - ニックネームの入力フィールド（1行テキスト、適切な最大文字数。例：20文字程度。プレースホルダー「ニックネームを入力」）。
  - 保存ボタン（ブランド色）。
  - 保存後は成功表示（トースト or インラインメッセージ）を出し、/member に戻る導線（戻るリンク or 自動遷移）を用意。
- 保存処理：
  - profiles テーブルに対して **upsert**（id = auth.uid() で。初回は insert、2回目以降は update になる）。nickname と updated_at を更新。
  - RLS により本人の行しか触れないので、id は必ずログインユーザーの uid を使う（クライアントから任意の id を渡さない）。
  - 保存方式（Server Action かクライアントから supabase.update か）は、既存の /mypage 等の保存パターンに合わせる。整合する形で実装する。

### 2. /member ダッシュボード側の更新（src/app/member/page.tsx）
- ログインユーザーの profiles からニックネームを取得する。
- 挨拶エリア：ニックネームが設定済みなら「ようこそ ○○さん」のように表示（今の「ようこそ／マイページ」の見せ方を活かしつつニックネームを反映）。未設定なら従来通り、またはニックネーム未設定が分かる控えめな表示。
- 「これからの機能」内の「プロフィール編集（準備中）」の枠を、**準備中ではなく /member/profile へのリンク**に変更する。点線・グレーピル（準備中）を外し、有効なカードにする。
  - ニックネーム未設定なら「ニックネームを設定」、設定済みなら「プロフィールを編集」のような文言にできるとなお良い（任意）。
- 残りの2枠（閲覧履歴・通知）は準備中のまま据え置き。

### 3. データ取得の注意
- /member と /member/profile はログインユーザー個別の内容なので ISR キャッシュ対象にしない（revalidate を付けない／動的のまま）。
- profiles の取得は、本人のセッションで RLS が効いた状態で行う（cookie レスの匿名クライアントは使わない。ここは認証済みクライアントを使う）。

## やってはいけないこと
- /mypage（オーナー用）に触れない。
- profiles の id にログインユーザー以外の値を入れない（RLSと整合させ、なりすまし防止）。
- 残り2枠（閲覧履歴・通知）の中身を実装しない。
- 既存の保存の仕組みやダッシュボードの他要素を壊さない。

## 完了後の検証
- `npx tsc --noEmit`：エラーなし。
- `npm run lint`：エラーなし。
- `npm run build`：成功。
- 未ログインで /member/profile にアクセス → /login にリダイレクトされる。
- ログイン状態で /member の「プロフィール編集」枠をタップ → /member/profile に遷移できる。
- /member/profile でニックネームを入力して保存 → 成功表示が出る。
- /member に戻ると挨拶エリアに「ようこそ ○○さん」のようにニックネームが反映されている。
- 一度保存した後に再度開くと、入力欄に保存済みニックネームがプリフィルされている（upsert の update 動作）。
- 別の会員でログインしても、他人のニックネームは見えない（RLSが効いている）。
- スマホ幅・PC幅でレイアウトが崩れない。
- /mypage・/saved・トップなど既存ページの挙動が変わっていない。
