# タスク：サロン詳細サブページ＋セラピスト詳細の表示速度改善（ISR化・並列化）

## 背景・症状
サロン詳細ページ（/salon/[id]）本体は最適化済み（ISR・cookieレス・並列化）で高速だが、そこから遷移する各ページ（サブページ群・セラピスト詳細）が**毎回遅い（体感3秒）**。2回目アクセスでも遅いため「初回生成」ではなく、**これらのページがISR化されておらず毎回フル動的描画されている**のが原因。トップ・サロン詳細本体でやったのと同じ最適化を、これらのページに適用する。

## 対象範囲
1. サロン詳細の全サブページ（`src/app/salon/[id]/` 配下の各 page.tsx）
   - price / coupon / news / info / diary / schedule / therapists / imasugu など、実在するサブページすべて。ディレクトリで実際の構成を確認すること。
2. セラピスト詳細ページ（`src/app/therapist/[id]/page.tsx` 等、実際のパスを確認）

## 前提（既存の仕組みを再利用）
- cookieを読まない匿名クライアント **createPublicClient** は作成済み。これを使う（場所不明なら createPublicClient で全体検索）。
- 公開データ（サロン・セラピスト・スケジュール・料金・クーポン・お知らせ・日記等）は匿名でSELECT可能（公開ページで既に表示されているため）。
- on-demand revalidate の仕組み（/api/revalidate、revalidateSalon ヘルパー等）は既存。サロン編集時に /salon/[id] 配下を 'layout' で一括無効化する仕組みは導入済み。サブページもこの配下なので既存の無効化でカバーされるはず（要確認）。
- ヘッダーの SavedSalonsMenu / AccountMenu / 通知ベルはクライアントコンポーネント。サーバー側でcookieを読む必要はない。
- next.config の images 設定は変更不要。

## ルール
- 見た目・表示内容は変えない。
- DBスキーマ変更なし。環境変数の変更なし。
- 完了後に型チェック・lint・build クリーンを確認し、feedback_auto_deploy.md の手順で自動コミット→プッシュ→デプロイまで進める。
- **既に最適化済みのトップ（/）・サロン詳細本体（/salon/[id]）のISRを絶対に壊さない。**

## 実装ステップ（対象の各ページに適用）

### 1. cookieレスクライアントへの置き換え
- 各ページが `@/app/lib/supabase/server` の createClient（cookie読む版）を使っているなら、cookieレスの **createPublicClient** に置き換える。
- これにより動的レンダリング強制が解除され、ISR化が可能になる。

### 2. revalidate の追加
- 各ページの先頭付近に `export const revalidate = 600;`（10分）を追加。
- サロン編集時の即時反映は、既存の revalidateSalon（/salon/[id] 配下を 'layout' で無効化）でサブページもカバーされることを確認する。セラピスト詳細は、セラピスト情報の保存箇所から該当の無効化を呼ぶ必要があれば追加する（既存の保存処理に合わせて。無ければ revalidate=600 の自然失効で許容）。

### 3. クエリの並列化
- 各ページのデータ取得を確認し、互いに依存しないクエリは Promise.all でまとめて並列実行する。
- ファイルごとに依存関係が違うので、各ファイルの実際の構造を読み取って判断（依存があるものは後段に残す。無理な並列化で結果を壊さない）。

### 4. 事前生成（generateStaticParams）— 判断して適用
- **全ページのISR化（上記1〜3）は必須。** これで「2回目以降が遅い」は解消する。
- generateStaticParams による事前生成は、**ビルド負荷とアクセス頻度のバランスを見て、適切なページにのみ付ける**こと：
  - セラピスト詳細（/therapist/[id]）：アクセスが多く初回の速さも重要なら、generateStaticParams で全セラピストIDを事前生成してよい（createPublicClient で therapists の全idを取得）。件数が極端に多くビルドが重くなる場合は見送り、ISR（初回生成・以降キャッシュ）で許容する。
  - サロン詳細サブページ（68サロン × 複数サブページ）：全組み合わせの事前生成はビルドが重くなるため、**原則 generateStaticParams は付けず、revalidate による「初回その場生成・以降キャッシュ」で十分**とする。
  - 上記は方針。実際の件数・ビルド時間を見て、過剰なら事前生成を絞る判断をすること。
- dynamicParams は true（デフォルト）のままにし、事前生成しないIDも初回アクセスで生成・キャッシュされるようにする（404にしない）。

## やってはいけないこと
- next.config を触らない。
- トップ・サロン詳細本体の既存ISRを壊さない。
- dynamicParams = false にしない。
- 認証が本当に必要なページ（もしあれば）まで cookieレス化しない。対象は公開ページのみ。サブページ・セラピスト詳細は公開情報なので対象でよいが、万一会員個別情報を出している箇所があればそこは動的のまま残す。
- 依存関係のあるクエリを無理に並列化して結果を壊さない。
- 表示内容・レイアウトを変えない。

## 完了後の検証
- `npx tsc --noEmit`：エラーなし。
- `npm run lint`：エラーなし。
- `npm run build`：成功。
  - **トップ（/）・サロン詳細本体（/salon/[id]）が引き続き ISR（○/●）を維持**（ƒ退行なし）。
  - サロン詳細サブページ・セラピスト詳細が ISR（revalidate）または事前生成（●）になっている（ƒ Dynamic のままになっていないこと）。
  - ビルド時間が極端に長くなっていないこと（事前生成しすぎていないか確認）。
- 本番反映後、サロン詳細から各サブページ（料金・クーポン・本日出勤・今すぐ・セラピスト一覧等）へ遷移し、**2回目以降が明確に速い**こと（毎回3秒の症状が解消）。
- サロン詳細からセラピスト詳細へ遷移し、同様に速いこと。
- 各ページの表示内容・レイアウトが以前と同じであること。
- オーナーがサロン情報・料金・クーポン等を編集 → 該当サブページに反映されること（既存の revalidateSalon でカバーされているか確認）。
