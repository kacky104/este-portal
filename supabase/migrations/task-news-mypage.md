# /mypage お知らせ管理UIの追加

/mypage に、店舗オーナーが自分の店舗のお知らせを管理（追加・編集・削除・公開/非公開切替）できるUIを追加したい。先日実装したクーポン管理（/mypage）と同じ作法・同じ設計に揃える。

## 認証・オーナー→サロンの紐付け
- 既存の /mypage の認証＆オーナー→サロン特定の仕組み（クーポン管理・今すぐ管理と同じ）をそのまま再利用する。
- すべての操作はログイン中オーナーの自店舗（salon_id）に対してのみ。他店舗のお知らせは操作・表示しない。

## /mypage のお知らせ管理セクション（タブ）
- 既存 /mypage のレイアウト/タブの作法（クーポン管理と同様）に合わせて「お知らせ」セクションを追加。
- 一覧：自店舗のお知らせを公開・非公開含めて表示（タイトル / 公開状態 / 公開日時）、published_at の新しい順。
- 新規追加フォーム：title（必須）/ content（必須・複数行入力）/ 公開・非公開トグル。published_at は作成時に now() で自動設定。
- 各お知らせの編集（同じ項目）・削除（確認あり）・公開/非公開のワンタップ切替。
- データ書き込みは既存 /mypage（クーポン管理）の方式に合わせる。

## RLS の更新（マイグレーションSQLを作成、実行はユーザーが Supabase で行う）
- announcements に、オーナーが自分の店舗のお知らせを INSERT/UPDATE/DELETE できるポリシーを追加。スコープは厳密に「その行の salon_id がログインオーナーの担当サロンと一致する場合のみ」。他店舗は不可。
- オーナーは自店舗のお知らせを公開・非公開に関わらず SELECT できるようにする（/mypage 一覧用）。公開ページ用の「is_published=true は誰でも SELECT」既存ポリシーは維持。
- クーポンのオーナー用RLS（coupons のオーナー用ポリシー）と同じ考え方・同じ書き方に揃える。既存の管理者UID用・公開SELECTポリシーは壊さない。

## スコープ/厳守
- 公開ページ /salon/[id]/news の表示は壊さない（読み取りは従来どおり、新しい順）。
- /mypage の既存機能（クーポン管理・今すぐ管理など）は一切壊さない。お知らせセクションの追加のみ。

## 進め方
作業前に、(a) /mypage のクーポン管理の実装（認証・オーナー特定・フォーム・データ書き込み）、(b) announcements の既存スキーマ/RLS、(c) coupons のオーナー用RLSの書き方、を確認し、それに合わせて実装すること。
完了したら feedback_auto_deploy.md に従って自動でコミット→プッシュ→デプロイまで進めてOK。RLS更新のマイグレーションSQLはユーザーが Supabase の SQL Editor で実行する。
