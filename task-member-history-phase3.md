# タスク：会員の閲覧履歴（フェーズ3）

## 目的
ログイン会員が見たサロン（/salon/[id]）・セラピスト（/therapist/[id]）を自動で記録し、専用ページ /member/history で「最近見た順」に振り返れるようにする。会員ダッシュボード（/member）の「閲覧履歴（準備中）」枠を、実際に動く機能に置き換える。

**フェーズ3のスコープ：閲覧の自動記録＋履歴表示ページ。** 通知・新着フォロー（フェーズ4）はやらない。

## 前提（重要）
- DBテーブル `public.view_history` は**作成済み**（手動SQL実行済み）。構成：
  - `id uuid`（PK, default gen_random_uuid()）
  - `user_id uuid not null`（auth.users(id), on delete cascade）
  - `item_type text not null`（'salon' or 'therapist' のみ。check制約あり）
  - `item_id bigint not null`（サロンID または セラピストID）
  - `viewed_at timestamptz not null default now()`
  - `unique (user_id, item_type, item_id)` ← 同じ対象は1行。再閲覧時は viewed_at を更新（upsert）。
  - インデックス：`(user_id, viewed_at desc)`
  - RLS有効。本人のみ select / insert / update / delete 可能（auth.uid() = user_id）。
- 会員ログインは既存の Supabase Auth。/member/history は**ログイン必須**（未ログインなら /login へリダイレクト）。
- **閲覧履歴はログイン会員のみの機能。未ログイン（匿名）ユーザーの閲覧は記録しない**（localStorage 等は使わない）。
- /mypage はオーナー専用。**一切触らない。**
- サロン詳細（/salon/[id]）とそのサブページ、トップは ISR キャッシュ化済み。**記録処理でこのキャッシュを壊さないこと（後述）。**
- ブランド色を踏襲（ロゴ・見出し＝オレンジ→マゼンタ #FB923C→#DB2777、サロン＝ピンク、セラピスト＝紫）。

## ルール
- 新規ページ追加・記録処理の追加。feedback_auto_deploy.md の手順で自動コミット→プッシュ→デプロイまで進める。
- DBスキーマ変更はしない（テーブルは作成済み）。環境変数の変更なし。
- 既存ページ（トップ・サロン詳細・/saved・/mypage・/member 等）の表示や ISR キャッシュの挙動を壊さない。

## 実装内容

### 1. 閲覧の記録（自動）
- サロン詳細（src/app/salon/[id]/page.tsx の表示）とセラピスト詳細（/therapist/[id] の表示）を会員が開いたときに、view_history へ記録する。
- **記録は必ずクライアント側から行う**（サーバーコンポーネントで記録しない）。理由：これらのページは ISR キャッシュ済みで、サーバー側に記録処理を置くとキャッシュと相性が悪く、キャッシュヒット時に記録が走らない。クライアントの useEffect 等でページ表示後に記録を送る方式にする。
  - 実装イメージ：小さなクライアントコンポーネント（例 `ViewHistoryLogger`）を作り、item_type と item_id を props で受け取り、マウント時に1回だけ記録する。これをサロン詳細・セラピスト詳細ページに設置する。
- 記録の中身：
  - ログイン中の会員のセッションを確認し、**ログインしている場合のみ**記録する。未ログインなら何もしない（エラーも出さない）。
  - view_history へ **upsert**（onConflict: 'user_id,item_type,item_id'）。viewed_at を now() に更新（再閲覧で日時が最新化される）。
  - user_id は必ずセッションのユーザーIDを使う（クライアントから任意値を渡さない。RLSと整合）。
  - 記録失敗はユーザー操作を妨げない（try/catch で握りつぶし、画面には影響させない）。
- パフォーマンス配慮：記録はページ表示の妨げにならないよう非同期で投げっぱなしにする（表示をブロックしない）。同一ページ内で多重記録しないよう、マウント時1回に限定する。

### 2. 閲覧履歴ページ /member/history（新規）
- 新規ファイル `src/app/member/history/page.tsx`。ログイン必須（未ログインは /login へ redirect）。
- 本人の view_history を viewed_at の降順で取得（最近見た順）。サロンとセラピストを両方扱う。
  - 取得した item_id をもとに、salons / therapists テーブルから表示用情報（名前・画像・エリア等）を取得する。サロン用とセラピスト用の取得は Promise.all で並列化。
  - 既に削除されたサロン/セラピストが履歴に残っている場合は、表示時にスキップする（落ちないように）。
- 表示：
  - 共通ヘッダーを流用。「閲覧履歴」の見出し。
  - サロンの履歴とセラピストの履歴を分けて表示（タブ切替 or セクション分け）。/saved の作りに寄せると一貫性が出る。
  - 各アイテムはタップでその詳細ページ（/salon/[id] または /therapist/[id]）へ遷移。
  - 履歴が0件のときは空状態（「まだ閲覧履歴がありません」等）を控えめに表示。
- （任意）「履歴を消す」導線：今回は必須ではないが、delete ポリシーがあるので、将来用に1件削除 or 全消去を付けてもよい。難しければ省略可。

### 3. /member ダッシュボード更新（src/app/member/page.tsx）
- 「これからの機能」内の「閲覧履歴（準備中）」枠を、**/member/history へのリンク**に変更する（点線・準備中ピルを除去し、有効カードにする）。文言は「閲覧履歴／見る →」等。
- 残る「通知・新着フォロー」枠は準備中のまま据え置き。
- ダッシュボード自体への履歴一覧の埋め込みは不要（専用ページ /member/history に集約する方針）。

### 4. データ取得の注意
- /member/history はログインユーザー個別の内容なので ISR キャッシュ対象にしない（revalidate を付けない／動的のまま）。
- view_history・profiles 等の会員データは認証済みクライアントで取得し、RLS を尊重する（cookie レスの匿名クライアントは使わない）。
- **サロン詳細・セラピスト詳細ページに記録用クライアントコンポーネントを足す際、これらのページの ISR（revalidate）設定や cookie レス化を壊さないこと。** 記録はあくまでクライアント側の追加であり、サーバー側のデータ取得方式は変更しない。

## やってはいけないこと
- /mypage（オーナー用）に触れない。
- サロン詳細・トップの ISR キャッシュ／cookie レス化を壊さない（記録はクライアント側追加のみ）。
- view_history の user_id にログインユーザー以外の値を入れない（なりすまし防止・RLSと整合）。
- 未ログインユーザーの閲覧を記録しない。
- 通知・新着フォロー（フェーズ4）の中身を実装しない。

## 完了後の検証
- `npx tsc --noEmit`：エラーなし。
- `npm run lint`：エラーなし。
- `npm run build`：成功。サロン詳細（/salon/[id]）が引き続き ISR（●/revalidate）のまま（ƒ Dynamic に退行していないこと）。/member/history は動的（ƒ）でよい。
- ログイン状態でサロン詳細・セラピスト詳細をいくつか開く → /member/history にそれらが「最近見た順」で表示される。
- 同じサロンをもう一度開く → 履歴の行が増えず、そのサロンが先頭（最新）に来る（upsert で viewed_at 更新）。
- 未ログイン状態でサロン詳細を開いても記録されず、エラーも出ない。
- /member の「閲覧履歴」枠が準備中ではなくリンクになっており、/member/history へ遷移できる。
- 別の会員でログインしても他人の履歴は見えない（RLS）。
- スマホ幅・PC幅でレイアウトが崩れない。
- トップ・サロン詳細の表示速度（ISR）が維持されている。
